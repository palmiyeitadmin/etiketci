# /plan-feature

## Purpose
Convert an approved feature spec into an implementation plan, task sequence, dependency map, and verification checklist.
Translate a specification into a strictly ordered, PR-sized action list tracked in `task.md`.
The expected outcome is a detailed `plan.md` and an updated `task.md`.

## When To Use
Use this workflow when:
- immediately after the user approves a `spec.md` generated by the `/spec-feature` workflow.

Do not use this workflow when:
- no approved specification exists for complex features
- addressing trivial bugs

## Inputs
Required inputs:
- `spec.md` (approved specification).
- `agent.md` and related context in `docs/`.

## Procedure
1. Review the approved `spec.md` and related context.
2. Identify the exact scope of the request and dependencies.
3. Setup Task Tracker: Create or update `task.md` with a hierarchical checklist of PR-sized implementation chunks, ordering DB -> API -> UI.
4. Write Implementation Details: In a `plan.md` artifact, draft the concrete files that will be touched, libraries needed, and exact validation tests to execute.
5. Present Plan: Request user review via `notify_user` with `plan.md` attached.

## Outputs
This workflow must produce:
- Updated `task.md`.
- New `plan.md` artifact.

The output must include:
- sequence of implementation tasks
- dependencies identified
- concrete files to be touched
- unit/integration test additions explicitly listed

## Validation Checks
Before considering this workflow complete, verify:
- the task list orders database schema changes before API endpoint changes.
- the task list orders API endpoint changes before Next.js UI consumption.
- the plan explicitly lists unit/integration test additions.
- output is consistent with project architecture and MVP boundaries.

## Stop Conditions
Stop and report if:
- required input (`spec.md`) is missing.
- the request would violate project guardrails.

## Escalation Conditions
Escalate explicitly when:
- the implementation requires a NuGet or NPM package not currently in the project.
- a new architecture decision is required.

## Project-Specific Guardrails
Always enforce:
- Epson-only PDF print flow constraints during planning.
- UnitConverter consistency implicitly via planned tests.
- DB Code-first migrations are planned properly.

Never:
- plan frontend consuming an API that isn't planned to be built or updated first.
- plan feature without testing tasks.

## Completion Criteria
This workflow is complete only when:
- `plan.md` and `task.md` are updated
- validation checks pass
- the user approves the Implementation Plan
- the next action (implementation) is obvious to the next agent
