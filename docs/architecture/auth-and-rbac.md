# Architecture: Authentication and RBAC

## 1. Local Development Authentication Mode
**Decision:** Real Entra Dev Tenant flow ONLY.
There is **no separate local-mock issuer or fake auth mode** in the PLMS codebase. 
- **Rule:** All developers must authenticate against a designated Microsoft Entra ID Tenant (e.g., a shared developer tenant or personal MS Developer tenant) defined in their `.env.local` and `appsettings.Development.json`.
- **Reasoning:** Maintaining parity between local and production environments prevents token format surprises, CORS mismatches, and claim mapping bugs. 
- **Strict Boundary:** "Fake auth" or mock JWT generation logic is strictly forbidden in the application code, even in development environments.

## 2. Source of Roles
**Decision:** Roles are sourced exclusively from **Entra ID App Roles**.
- **Mechanism:** The Microsoft Entra ID App Registration defines the allowed App Roles. Users or Groups are assigned to these roles in the Azure Portal. The `roles` claim is injected directly into the JWT Access Token issued by Entra ID.
- **Mapping:** The Next.js frontend and .NET backend read the `roles` array directly from the token. There is **no database-side role mapping table** (no `AppUserRole` junction tables in Postgres).
- **Core Roles:** 
  - `Admin`: Full access. Treated as a superset role in backend policies.
  - `Reviewer`: Can review/approve templates.
  - `Operator`: Authorized to execute print jobs.
  - `Viewer`: Read-only access.
- **Multiple Roles:** Yes, a user may carry multiple roles in their token.
- **Missing Roles:** If no recognized role is present in the `roles` claim, the user is treated as anonymous for protected endpoints and gracefully falls back to the `Viewer` role in the UI for safe rendering.

## 3. Frontend Token Forwarding Strategy
**Decision:** Direct Browser -> External API using Client-Side Session.
- **Acquisition:** Next.js uses NextAuth (`next-auth/providers/azure-ad`) to execute the OAuth flow. NextAuth receives the Access Token and stores it in the encrypted NextAuth session.
- **Client Availability:** The Next.js session callback exposes the `accessToken` to the client-side `useSession()` hook.
- **Forwarding:** The frontend `api-client.ts` retrieves the `accessToken` from the NextAuth session and attaches it to the `Authorization: Bearer <token>` header for all outgoing requests.
- **CORS Boundary:** The browser makes direct requests to the `.NET 8` API (e.g., `http://localhost:8080/api/...`). Strict CORS is configured in the backend to allow only the Next.js origin (`http://localhost:3000` locally). There is no Next.js API proxy layer for business data.
- **Refresh/Re-auth:** When the access token expires, API requests will return `401 Unauthorized`. The frontend API client must catch this and redirect the user back to `/api/auth/signin` to refresh the session interactively.

## 4. Audit & Auth Interaction
**Decision:** Centralized Audit Enrichment.
- **Requirement:** All privileged or mutating endpoints (Create, Update, Delete, Print) MUST be auditable.
- **Correlation:** Every mutation request must carry a `x-correlation-id` header (generated by the frontend API client).
- **Enrichment:** The backend `ApplicationDbContext` or centralized middleware is responsible for extracting the `UserId` (from `ClaimsPrincipal.Name`), `roles`, and `CorrelationId` (from HTTP headers) and injecting them into any generated `AuditLog` records. 
- **Rule:** Controllers should NOT manually construct audit logs for domain mutations. The persistence layer must intercept and log inherently. Future print/reprint auditing will utilize this same immutable ledger.

## 5. Auth Smoke Test Matrix
To guarantee the RBAC boundary operates securely, validate the following matrix:

| Scenario | Simulated Action | Expected Result | Reason |
| :--- | :--- | :--- | :--- |
| **Anonymous** | Request to `[Authorize]` endpoint without Bearer token. | `401 Unauthorized` | No token provided. |
| **Missing Role** | Request to `RequireOperator` endpoint with only `Viewer` role. | `403 Forbidden` | Valid token, insufficient claims. |
| **Admin Superset** | Request to `RequireOperator` endpoint with `Admin` role. | `200 OK` | Admin policy overrides explicitly. |
| **Expired Token** | Request to protected endpoint with expired JWT. | `401 Unauthorized` | Token lifetime exceeded. |
| **Invalid Issuer** | Request with JWT signed by a different tenant. | `401 Unauthorized` | JWKS validation failure. |
| **CORS Mismatch** | Request from `https://evil.local` to API. | Browser Preflight Block | Origin not in allowed CORS list. |
