# Palmiye Label Management System (PLMS)
**Agentic Context & Constraints**

This document establishes the foundational rules, context, and guardrails for all future AI agents working on this repository. **Do not deviate from these constraints unless explicitly instructed to do so by the user.**

## 1. Project Purpose & Product Scope
PLMS is a web-based label management and printing system designed to standardize how labels are created, managed, and printed across the organization.

## 2. MVP Boundaries (CRITICAL)
The MVP scope is strictly limited to ensure a successful initial delivery.
*   **Epson-Only MVP:** The only supported printer target is the Epson ColorWorks CW-C4000e.
*   **NO QZ Tray:** Do not introduce QZ Tray or any client-side direct-to-port printing middleware in the MVP.
*   **NO Zebra/ZPL:** Do not add Zebra support or RAW ZPL generation. If designing abstractions, do not leak ZPL constraints into the core model.
*   **PDF Print Flow Only:** The backend generates a fixed-size PDF. The frontend opens this PDF in the browser. The user manually prints using the OS/Epson Windows driver.
*   **User-Confirmed Print Status:** The system relies on the user to confirm whether a print was successful or failed via the UI.
*   **Basic Editor Only:** Avoid advanced Figma-like features (auto layout, nested groups). The editor supports: text, rectangle, line, barcode, QR, image, basic zoom/pan/select, layer list, property panel, snap-to-grid, and undo/redo.
*   **Centralized Unit Conversion:** All conversions (mm to dpi/dot/pixel) must go through a central `UnitConverter`.
*   **Canonical Validation:** Import validation must perform dry-runs before committing.

## 3. Architecture Summary & Tech Stack
*   **Frontend:** Next.js 14+ with TypeScript.
*   **Backend:** .NET 8 Web API.
*   **Database:** PostgreSQL (with EF Core).
*   **Infra:** Docker Compose for local development.
*   **Auth:** Microsoft Entra ID SSO.
*   **Rendering Pipeline:** Canonical JSON model -> Backend PDF Output & Frontend Preview.

## 4. Key Business Rules
*   **Template Lifecycle:** Draft -> In Review -> Approved -> Published -> Deprecated -> Archived. Templates are versioned and immutable once published.
*   **Audit Logging:** All critical actions (creation, state changes, printing, reprinting, data imports) must be audit-logged with correlation IDs and the associated user.
*   **Reprint Control:** Reprints must be explicitly governed and logged.
*   **Role-Based Access Control (RBAC):** All API endpoints must enforce strict permissions based on Entra ID roles.

## 5. Design Principles
*   **Single Source of Truth:** The PDF generated by the backend is the definitive representation of the label. The frontend preview must match it visually.
*   **Idempotency:** Print jobs and import operations must be safely retriable.
*   **Fail Fast:** Validation should happen early (e.g., dry-run for CSV imports).

## 6. Repository Conventions & Coding Standards
*   **C# / .NET:** Follow Clean Architecture principles. Use Repository/Service patterns. Keep controllers thin. Use minimal APIs or controllers based on team preference, but enforce standard DTOs.
*   **TypeScript / Next.js:** Strict mode enabled. Use functional components, hooks, and standard Tailwind/CSS Modules. No any types.
*   **Database:** EF Core Code-First migrations exclusively.
*   **Commits:** Use semantic commit messages (`feat: ...`, `fix: ...`).

## 7. Execution & Definition of Done (DoD)
Before any task or implementation is considered complete, it must satisfy:
*   **Testing:** Passing unit tests for core logical changes (such as the `UnitConverter` or Canonical model parsers).
*   **Visual E2E Parity:** Exact visual verification of PDF dimension boundaries against the Next.js UI preview where applicable.
*   **Audit Compliance:** Correct Audit logs inserted and verified for all mutative operations (print flows, reprints, state changes, DB imports).
*   **Guardrail Check:** No violation of the Epson-only constraint. No ZPL, no QZ Tray, no silent background printing added.
*   **Integration tests:** EF Core queries verified via real Postgres Testcontainers.
